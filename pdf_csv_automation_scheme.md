# PDF→CSV フォーマット自動化スキーム設計書

## 1. 課題の整理

添付のCSA質問票（266ページ）を確認したところ、以下の特性が確認できました。

- **構造の多様性**: テーブル、チェックボックス（✓ / ❍）、自由記述、URLリファレンス、入力フィールド（`_ _ _`）が混在
- **テーブル検出の不安定さ**: pdfplumberでPage 11は16テーブルを検出するが、セル結合や空フィールドの処理が不正確
- **テキスト抽出の断片化**: 改行位置が元のレイアウトと一致しない、テキストボックスがフラットテキストとして抽出される
- **様式の非統一性**: クライアント・年度・評価機関ごとにフォーマットが異なる

これらを踏まえ、「完全自動化」ではなく**「段階的にルールを蓄積し、人間チェックの負荷を漸減させる」**アーキテクチャを提案します。

## 3. 各ステージの詳細設計

### Stage 1: 前処理 — 構造抽出・正規化

**目的**: PDFから可能な限り構造化された情報を取り出す

**アプローチ**: 複数の抽出エンジンを併用し、最良結果を採用する

| 抽出対象 | 推奨ツール | 備考 |
|---------|-----------|------|
| テーブル | Camelot (lattice + stream) | 罫線ベースと空白ベースの両方を試行 |
| テキスト | PyMuPDF (fitz) | ブロック単位でレイアウト情報を保持 |
| チェックボックス | 正規表現 + Unicode検出 | ✓ / ❍ / ☑ / ☐ のパターンマッチ |
| フォーム構造 | pdfplumber + カスタムロジック | フィールドラベルと値の対応検出 |

**出力形式**: 中間JSON（ページ単位）

```json
{
  "page": 11,
  "elements": [
    {
      "type": "table",
      "confidence": 0.85,
      "extraction_method": "camelot_lattice",
      "headers": ["Aggregate", "Revenue", "CapEx", "OpEx"],
      "rows": [...]
    },
    {
      "type": "text_block",
      "content": "Question Rationale...",
      "bounding_box": [x0, y0, x1, y1]
    },
    {
      "type": "checkbox_group",
      "items": [
        {"label": "Option A", "checked": true},
        {"label": "Option B", "checked": false}
      ]
    }
  ]
}
```

### Stage 2: 分類 — 様式パターン判定

**目的**: 入力ファイルがどの「様式パターン」に該当するか判定する

**二段構えの分類ロジック**:

1. **ルールベース判定（優先）**: ファイル名・ヘッダーテキスト・特徴的キーワード・ページ構造のフィンガープリントで判定
   - 例: "CSA 2024" + セクション番号パターン → S&P CSA 2024形式
   - 例: "CDP Climate Change" + モジュール構造 → CDP回答形式

2. **LLMフォールバック（未知様式のみ）**: 最初の数ページを抽出し、Claude APIに「この文書はどのような様式か？どのようなCSV構造に変換すべきか？」を問い合わせ

**パターンDB構造**:

```json
{
  "pattern_id": "csa_2024",
  "name": "S&P Global CSA 2024",
  "fingerprint": {
    "header_keywords": ["CSA 2024"],
    "section_pattern": "^\\d+\\.\\d+",
    "page_structure": "header + question + response_fields"
  },
  "mapping_template": "csa_2024_mapping.json",
  "last_updated": "2025-03-06",
  "accuracy_score": 0.92
}
```

### Stage 3: 変換 — CSV構造マッピング

**目的**: 抽出データを目的のCSV構造にマッピングする

**マッピングルール例（CSA質問票の場合）**:

```json
{
  "output_columns": [
    "section_id",
    "question_id", 
    "question_text",
    "response_type",
    "response_value",
    "supporting_url",
    "page_reference"
  ],
  "extraction_rules": [
    {
      "field": "section_id",
      "source": "text_block",
      "pattern": "^(\\d+\\.\\d+)"
    },
    {
      "field": "response_type",
      "logic": "detect_from_element_type",
      "mapping": {
        "checkbox_group": "multiple_choice",
        "table": "tabular_data",
        "text_block_after_question": "free_text"
      }
    }
  ]
}
```

### Stage 4: 検証 — 品質チェック

**自動バリデーション項目**:

| チェック項目 | 内容 | 信頼度への影響 |
|------------|------|-------------|
| 型整合性 | 数値フィールドに文字列が入っていないか | 高 |
| 欠損チェック | 必須フィールドが空でないか | 高 |
| 範囲チェック | パーセンテージが0-100の範囲か | 中 |
| 参照整合性 | URLが有効か、ページ参照が文書内に存在するか | 低 |
| 前年比較 | 過去データと大幅に乖離していないか | 中 |

**信頼度スコアの算出**:

各セルに対して 0.0〜1.0 の信頼度を付与。

- 1.0: ルールベースで完全マッチ、バリデーション全通過
- 0.7-0.9: 抽出は成功したが一部曖昧さあり
- 0.5-0.7: LLMフォールバックで変換、要確認
- 0.5未満: 抽出失敗または大きな不整合あり

### Stage 5: 人間レビュー — 選択的レビュー

**レビューUIの設計思想**: 信頼度スコアに基づき、人間がチェックすべき箇所だけをハイライト表示する

```
レビュー画面イメージ:
┌──────────────────────────────────────────────┐
│ [PDF原本表示]          │ [CSV変換結果]         │
│                        │                       │
│  ← 該当箇所を         │ セルの背景色で         │
│     ハイライト表示      │ 信頼度を可視化         │
│                        │ 🟢 高信頼（チェック不要）│
│                        │ 🟡 中信頼（目視推奨）   │
│                        │ 🔴 低信頼（要確認）     │
└──────────────────────────────────────────────┘
```

**段階的な省力化の仕組み**:

- 初期: 全セルレビュー（ただしハイライトで優先順位がつく）
- 蓄積期: 高信頼セルのスキップを許可（サンプリングチェックに移行）
- 成熟期: 低信頼セルのみレビュー、高信頼は自動承認

### Stage 6: 学習蓄積 — Claude Codeによるルール自動生成

**ここが本スキームの核心です。**

人間レビューで修正が入った場合、その修正内容を元にClaude Codeが新しいルールを生成し、ルールDBに追加します。

**フロー**:

```
[人間が修正] → [修正差分を記録] → [Claude Codeに投入]
                                        │
                                        ▼
                              「この修正パターンを
                               汎用ルール化してください」
                                        │
                                        ▼
                              [ルール候補コード生成]
                                        │
                                        ▼
                              [既存テストケースで検証]
                                        │
                                        ▼
                              [合格 → ルールDB追加]
                              [不合格 → 人間確認]
```

**Claude Codeへの指示例**:

```
以下の修正履歴を分析し、再利用可能な抽出ルールを
Pythonコードとして生成してください。

【修正前】
CSA質問票 Page 101, Table 0:
  抽出結果: ["Energy Efficiency", "Load Optimization\nthe maximization..."]
  → セル内改行がテーブル境界と誤認された

【修正後】
  正解: ["Energy Efficiency | Load Optimization", "the maximization..."]
  → 「Energy Efficiency」と「Load Optimization」は同一行の隣接セル

【要求】
- この修正を汎用化するルールを生成
- 同様のパターン（CSAのクロス集計テーブル）に適用可能にする
- テストケースも生成する
```

## 4. 技術スタック案

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| PDF抽出 | PyMuPDF + Camelot + pdfplumber | 複数エンジン併用で精度最大化 |
| 中間データ | JSON（ページ単位） | 柔軟な構造表現、デバッグ容易 |
| ルールエンジン | Python (カスタム) | ルールの追加・修正が容易 |
| LLMフォールバック | Claude API (Sonnet) | コスト効率と精度のバランス |
| ルール自動生成 | Claude Code | 修正差分からのコード生成 |
| ルールDB | JSON/YAML ファイル（Git管理） | バージョン管理、差分追跡が容易 |
| レビューUI | React (Web) | PDF並列表示、インタラクティブ操作 |
| バリデーション | Python + pytest | ルール生成時の自動テスト |

## 5. 導入ロードマップ

### Phase 1: 基盤構築（1-2ヶ月）

- PDF抽出パイプラインの構築（Stage 1）
- 主要様式3-5パターンのマッピングルール手動作成（Stage 2-3）
- 基本バリデーションの実装（Stage 4）
- **KPI**: 主要様式で抽出精度70%以上

### Phase 2: レビューUI + 学習ループ（2-3ヶ月）

- レビューUI構築（Stage 5）
- 修正差分記録の仕組み構築
- Claude Codeによるルール生成パイプライン構築（Stage 6）
- **KPI**: レビュー時間を手作業比50%削減

### Phase 3: 精度向上 + 自動化拡大（3-6ヶ月）

- ルール蓄積による精度向上
- 高信頼セルの自動承認開始
- 新規様式の半自動対応
- **KPI**: 高信頼セル率80%以上、レビュー時間を手作業比80%削減

### Phase 4: 運用最適化（6ヶ月〜）

- サンプリングチェックへの移行
- 異常検知の高度化
- 新規様式の完全自動分類
- **KPI**: 新規様式でも初回精度60%以上

## 6. リスクと対策

| リスク | 影響 | 対策 |
|-------|------|------|
| LLMハルシネーション | 誤ったデータがシステムに入る | LLMは分類・ルール生成にのみ使用し、データ変換自体はルールベースで実行。LLM出力は必ずテストケースで検証 |
| PDF構造の想定外変化 | 抽出失敗 | 信頼度スコアで自動検知し、人間レビューにフォールバック |
| ルールの競合・退行 | 既存パターンの精度低下 | Git管理 + 回帰テストで品質担保 |
| 初期投資の回収遅延 | ROI悪化 | Phase 1で最頻出の様式から着手し、早期に効果を実感できる設計 |

## 7. estomaDB との連携ポイント

このスキームはestomaDBの既存アーキテクチャと以下の点で整合します。

- **ルールベース優先・LLMフォールバック**: estomaDBと同一の設計思想
- **ナレッジ管理レイヤー**: コンサルティング業務で得た知見をルールとして蓄積する仕組みが共通
- **段階的自動化**: 一気に完全自動化を目指さず、人間の判断を組み込みながら精度を向上させるアプローチ

差異としては、estomaDBが「どこからデータを取るか」のクローリング・収集に重点を置くのに対し、本スキームは「取得済みデータをどう構造化するか」の変換・整形に特化している点です。両者はパイプラインの前段・後段として自然に接続できます。
